## Установка

### Вариант А: Локальный запуск

#### 1. Клонируйте репозиторий

```bash
git clone https://github.com/Steev193/hh-ru-apply.git
cd hh-ru-apply
```

#### 2. Установите зависимости

```bash
npm install
```

#### 3. Установите браузер Playwright

```bash
npx playwright install chromium
```

#### 4. Скопируйте переменные окружения

```bash
cp .env.example .env
```

### Вариант Б: Docker

#### 1. Клонируйте репозиторий

```bash
git clone https://github.com/Steev193/hh-ru-apply.git
cd hh-ru-apply
```

#### 2. Скопируйте переменные окружения

```bash
cp .env.example .env
```

#### 3. Запуск дашборда

```bash
docker compose up -d
```

Дашборд будет доступен на http://127.0.0.1:3849. Данные сохраняются в локальную папку `data/`.

#### 4. Запуск других скриптов

Для входа:
```bash
docker compose run --rm dashboard npm run login
```

Для сбора вакансий:
```bash
docker compose run --rm dashboard npm run vacancies
```

Для оценки:
```bash
docker compose run --rm dashboard npm run harvest
```

Для отклика:
```bash
docker compose run --rm dashboard npm run hh-fill-letter -- --id=<uuid>
```

## Первый запуск и вход

```bash
npm run login
```

**Что произойдёт:**

1. Откроется окно Chromium
2. Перейдёт на hh.ru
3. Вам нужно **войти в свой аккаунт** — ввести логин, пароль, пройти капчу и двухфакторную аутентификацию, если она включена
4. После успешного входа вернитесь в терминал и нажмите **Enter**

**Важно:**

- Браузерный профиль сохраняется в `data/session/chromium-profile`
- При последующих запусках логиниться **не нужно** — скрипты переиспользуют тот же профиль
- Сессия со временем «протухает» — если скрипт ругается на авторизацию, запустите `npm run login` снова

## Настройка поиска

Откройте `config/search-keywords.txt` и напишите поисковые запросы, по одному на строку:

```
python backend разработчик
senior python developer москва удалённо
python data engineer
django backend
```

**Советы:**

- Чем конкретнее запрос, тем релевантнее результат
- Добавьте город и тип работы, если важно
- Не задавайте слишком общие запросы вроде просто `python` — будет много мусора

Настройки лимитов и пауз находятся в `.env`:

```bash
HH_PER_KEYWORD_LIMIT=8        # сколько вакансий брать по каждому ключу
HH_MAX_TOTAL=7                # максимум вакансий за запуск
HH_OPEN_DELAY_MIN_MS=3000     # минимальная пауза между открытиями
HH_OPEN_DELAY_MAX_MS=5000     # максимальная пауза между открытиями
```

Увеличивая `HH_MAX_TOTAL`, можно собрать больше вакансий за один запуск.

## Добавление резюме

Создайте папку `CV/` в корне проекта и положите туда ваши резюме (.md, .txt или .pdf).

**Формат `.md` (рекомендуется):**

```markdown
# Имя Фамилия

## Опыт
Компания: Senior Python Developer
- Разработка микросервисов...
- ...

## Навыки
Python, FastAPI, PostgreSQL, docker, kubernetes
```

Файлы из `CV/` **не попадают** в git — папка добавлена в `.gitignore`.

Эти резюме используются скриптом `harvest` при оценке — LLM сравнивает текст резюме с требованиями вакансии и даёт скор совпадения.

## Настройка AI для оценки вакансий

### Polza AI (рекомендуется)

1. Зарегистрируйтесь на [polza.ai](https://polza.ai/), создайте API key.
2. В **`.env`** в корне (шаблон — `.env.example`):

```
POLZA_API_KEY=...
```

Опционально `POLZA_MODEL`, `LLM_PROVIDER` — см. `config/OPENROUTER.md`.

### OpenRouter (альтернатива)

1. Зарегистрируйтесь на [openrouter.ai](https://openrouter.ai/)
2. Создайте API key
3. Создайте файл **`.env`** в корне проекта (можно скопировать из `.env.example`) и запишите туда:

```
OpenRouter_API_KEY=sk-or-v1-ВАШ_КЛЮЧ
```

Этот файл игнорируется в git и не попадёт в репозиторий.

**Модели (OpenRouter):**

По умолчанию используется бесплатная модель `qwen/qwen3.6-plus-preview:free`.

Если хотите разрешить платные модели, добавьте в `.env`:

```bash
OPENROUTER_ALLOW_PAID=1
```

Более подробно — в `config/OPENROUTER.md`.

## Шаблон сопроводительного письма

1. Скопируйте шаблон:

```bash
cp config/cover-letter.example.txt config/cover-letter.txt
```

2. Отредактируйте `config/cover-letter.txt` — напишите, как вы обычно представляете себя: что важно упомянуть, какой стиль, какие формулировки.

Этот файл — **шаблон-затравка** для генерации писем. LLM будет использовать его как основу и адаптировать под конкретную вакансию.

**Эталоны стиля (опционально):**

Скопируйте примеры ваших реальных писем:

```bash
cp config/cover-letter-style-examples.example.txt config/cover-letter-style-examples.txt
```

Разделяйте несколько примеров строкой `---`. Это помогает LLM имитировать вашу манеру, а не генерировать «нейросетевой» текст.

## Сбор вакансий

```bash
npm run vacancies
```

Если параллельно создан **флаг остановки** (кнопка «Остановить поиск» в дашборде во время `harvest`), тот же флаг прерывает `vacancies` перед следующим ключом и перед открытием следующей вкладки вакансии.

**Что произойдёт:**

1. Откроется браузер с вашей сессией hh.ru
2. По очереди выполняются запросы из `config/search-keywords.txt`
3. Страницы вакансий сохраняются для дальнейшей обработки

```bash
npm run harvest
```

**Что произойдёт:**

1. Скрипт загружает собранные страницы вакансий
2. Для каждой вакансии отправляется запрос к OpenRouter с:
   - Текстом вакансии
   - Вашими резюме из `CV/`
3. Получает оценки (модель + при необходимости детерминированные оси):
   - **scoreVacancy**, **scoreCvMatch** — как раньше
   - **scoreWorkFormat**, **scoreLocation** — формат работы и локация относительно блоков `scoringWorkFormat` / `scoringGeo` в `preferences.json` (если в `llmScoreWeights` заданы ненулевые веса `workFormat` и `location`, итог **scoreOverall** считается как взвешенная сумма четырёх осей; иначе — прежняя логика двух осей или `scoreOverall` от модели)
   - опционально мягкий бонус по зарплате (`scoringSalarySoft`) и правки по формату (`useRuleBasedAdjustment`)
4. Результаты сохраняются в `data/vacancies-queue.json` (в т.ч. `scoreSortKey` для сортировки)

Оценки отфильтровывают очевидно неподходящие вакансии и помогают приоритезировать.

## Оценка через LLM

Скрипт `harvest` делает это автоматически. Но если хотите перезапустить оценку с новыми简历 или весами:

```bash
npm run harvest
```

**Веса оценок** настраиваются в `config/preferences.json`:

```json
{
  "llmScoreWeights": {
    "vacancy": 0.25,
    "cvMatch": 0.45,
    "workFormat": 0.15,
    "location": 0.15
  },
  "scoringGeo": {
    "baseCity": "Казань",
    "acceptableCities": [],
    "remoteIsAcceptable": true,
    "penaltyUnknownLocation": -3
  },
  "scoringWorkFormat": {
    "prefer": "remote_first",
    "remoteBonus": 0,
    "hybridBonus": 0,
    "officePenalty": 0,
    "useRuleBasedAdjustment": false
  },
  "scoringSalarySoft": {
    "enabled": false,
    "maxBonusPoints": 8,
    "referenceUsd": 4000
  }
}
```

Сумма весов в `llmScoreWeights` нормализуется к 1. Если `workFormat` и `location` равны 0, поведение как у пары только `vacancy` / `cvMatch`.

## Работа с дашбордом

```bash
npm run dashboard
```

Откроется http://127.0.0.1:3849

### Сбор вакансий из дашборда

Первая вкладка **«Сбор вакансий»** открывает форму с переменными `harvest` (`HH_PER_KEYWORD_LIMIT`, `HH_SESSION_LIMIT`, `HH_MAX_TOTAL`, паузы и джиттеры) плюс **логика ключей** и **рабочий интервал**. Значения по умолчанию читаются с сервера из окружения (обычно из `.env`); **заполненные поля формы передаются в дочерний процесс и имеют приоритет**.

- `HH_KEYWORDS_LOGIC`: **`loop`** — без остановки повторять проходы по полному списку ключей (остановка — кнопка **«Остановить поиск»** в дашборде или завершить процесс `harvest`); **`cycles`** — `HH_KEYWORDS_CYCLES` полных проходов; **`keywords`** — один проход только по первым `HH_KEYWORDS_MAX` строкам файла (пусто = весь файл).
- `HH_WORK_HOURS_ENABLED=1` — только в интервале `HH_WORK_HOUR_START`–`HH_WORK_HOUR_END` (целые часы 0–23, **включительно**, локальное время машины) выполняются поиск и открытие карточек; иначе процесс ждёт (событие `idle_work_hours` в логе).

Кнопка **«Старт поиска»** запускает **`harvest.mjs`** в фоне (не блокирует переключение на «На проверке» и др.). Пока активен сбор, рядом показывается **«Остановить поиск»**: запрос уходит в `POST /api/harvest-stop-graceful`, создаётся флаг остановки (абсолютный путь; относительный `HH_DATA_DIR` в `.env` считается от **корня репозитория**). При запуске из дашборда дочернему процессу передаётся `HH_GRACEFUL_STOP_FILE` с тем же путём, что и у `POST /api/harvest-stop-graceful`, чтобы не было расхождений с `cwd`. Процесс **не убивается** — после полной обработки текущей карточки (или после `keyword_done` в фазе поиска, без опроса следующих ключей) прогон завершается штатно. Та же кнопка, если процесс уже не жив, а в логе остался незакрытый прогон (есть `HARVEST_RUN` без строки `harvest exit`), **дописывает завершение в лог** — раньше для этого была отдельная операция «сбросить зависшее состояние». Для очень старых логов без PID в маркере браузер запросит подтверждение и отправит повторный запрос с `{"force":true}` (аналог `POST /api/harvest-clear-stale`). В `GET /api/harvest-status` поле **`gracefulStopPending`** показывает, что остановка уже запрошена. В лог дописывается маркер `======== HARVEST_RUN … pid=… ========`; если **PID в последнем маркере не жив**, прогон **не** считается активным (после перезапуска дашборда не остаётся «вечной» блокировки без процесса). Статистика и список открытых URL обновляются по опросу API; полный вывод — в `data/harvest-run.log`. Индикатор **«Идёт сбор вакансий…»** и поле **`running`** в API учитывают лог: при расхождении с внутренним флагом процесса (часто Windows + `detached` у spawn) прогон считается активным, пока после маркера `runId` нет строки `--- harvest exit code=`. Счётчики и списки ключей парсятся из **последнего** прогона: окно до ~48 MiB от конца файла, начиная с последней строки `======== HARVEST_RUN … ========`, чтобы длинный обход карточек не вытеснял из окна события `keyword_done` / `keyword_active` фазы поиска. **Текущее ключевое слово** — первая строка блока статистики и последнее `keyword_active` в этом окне. Второй запуск, пока первый не завершился, вернёт ошибку 409.

После **успешного** завершения каждого прохода (без кооперативной остановки) первая непустая строка в `config/search-keywords.txt` переносится в конец. Отключить ротацию: `HH_ROTATE_KEYWORD_AFTER_RUN=0` в `.env`.

### Интерфейс

Дашборд содержит несколько вкладок:

#### Сбор вакансий

Настройки лимитов и задержек, старт `harvest`, остановка (включая сброс зависшего индикатора в логе), счётчики и раскрывающийся список URL.

#### Vacancies — очередь ваканций

Карточка на всю ширину окна (с небольшими отступами): слева — оценка, ссылка, комментарии модели и действия; в центре — **условия работы** (блок с hh.ru: доход, опыт, занятость, оформление, график и т.д.); справа — **полное описание вакансии** с страницы hh.ru (прокрутка при длинном тексте).

Поля `hhWorkConditions` и `vacancyDescriptionFull` заполняются при `npm run harvest` и при нажатии на карточке **обновить** (иконка ↻ рядом с «Создать сопроводительное»), которая вызывает `POST /api/vacancy/refresh-body` и заново тянет страницу через Playwright.

Старые записи в `data/vacancies-queue.json` без этих полей покажут подсказку «Нет данных» до обновления.

Здесь отображаются вакансии с оценками. Можно:

- Переключать вкладки: на проверке / подходят / отклонённые
- Просматривать детали вакансии и удалять запись из очереди (×)

#### Письма — генерация и редактирование сопроводительных

1. Выберите вакансию из списка
2. Нажмите «Сгенерировать» — LLM создаст письмо на основе:
   - Текста вакансии
   - Вашего шаблона (`config/cover-letter.txt`)
   - Ваших примеров стиля (если есть)
3. Отредактируйте письмо, если нужно
4. Нажмите «Подходит» в модальном окне — письмо сохраняется, вакансия переходит во вкладку «Подходят»

Сохранённые таким образом письма подмешиваются как эталоны при будущих генерациях.

#### Settings — настройки

Порт сервера, лимиты и прочие параметры.

## Отклик на вакансию

Есть два способа.

### Способ 1: Через форму отклика на странице вакансии (`hh-fill-letter`)

Открывает страницу, нажимает «Откликнуться», вставляет письмо и ждёт ручной проверки.

```bash
npm run hh-fill-letter -- --id=<uuid-записи-из-очереди>
```

uuid берётся из `data/vacancies-queue.json` (поле `id` у нужной записи).

Или по URL:

```bash
npm run hh-fill-letter -- --url=https://hh.ru/vacancy/123456 --text-file=./letter.txt
```

**Флаги:**

- `--stay-open` — не закрывать браузер (удобно для отладки)

### Способ 2: Через чат с работодателем (`hh-apply-chat`)

Письмо уходит **в переписку с работодателем**, а не в поле формы на карточке вакансии.

```bash
npm run hh-apply-chat -- --id=<uuid-записи-из-очереди>
```

Флаги:
- `--stay-open` — не закрывать браузер
- `--dry-run` — открыть чат, но не вставлять письмо
- `--no-submit` — открыть форму отклика, но не отправлять

### Способ 3: Из дашборда

На карточке вакансии в дашборде есть кнопка «Отклик в браузере» (доступна при утверждённом письме). Она запускает `hh-apply-chat` на той же машине, где запущен сервер дашборда.

## Обновление селекторов при смене вёрстки

Если при запуске скриптов появляются ошибки вроде «element not found» или selector не найден — hh.ru, возможно, изменил вёрстку.

1. Запустите генерацию селекторов:

```bash
npm run codegen-hh
```

2. Если не помогло, запустите вручную:

```bash
npx playwright codegen https://hh.ru
```

Откроется окно, в котором можно кликать по элементам и сразу получать селекторы. Скопируйте новые селекторы в:
- `lib/hh-response-selectors.mjs` — для формы отклика
- `lib/hh-chat-selectors.mjs` — для чата

## Решение типичных проблем

| Проблема | Решение |
|----------|---------|
| Скрипт не авторизуется на hh.ru | Запустите `npm run login` для обновления сессии |
| «Element not found» при отклике | Обновите селекторы через `npm run codegen-hh` или вручную |
| Скрипт зависает на открытии вакансии | Увеличьте `HH_OPEN_DELAY_MAX_MS` в `.env` |
| LLM даёт странные оценки | Проверьте CV в `CV/` — они должны быть читаемыми и подробными |
| Дашборд не запускается | Проверьте, свободен ли порт 3849, или задайте `DASHBOARD_PORT` |
| Письмо генерируется слишком длинным | Уменьшите `COVER_LETTER_STYLE_MAX_CHARS` в `.env` |
| Скрипт запускается, но ничего не делает | Проверьте `config/search-keywords.txt` — в нем должны быть запросы |

## Порядок действий для типичного сценария

Вот стандартный рабочий цикл:

1. **Один раз:** `npm run login` — войти в аккаунт
2. **Подготовка:** написать `config/search-keywords.txt`, резюме в `CV/`, шаблон письма в `config/cover-letter.txt`
3. **Сбор:** `npm run vacancies` — собрать вакансии
4. **Оценка:** `npm run harvest` — оценить через LLM
5. **Дашборд:** `npm run dashboard` — перейти в http://127.0.0.1:3849
6. **Письма:** для интересующих вакансий — «Создать сопроводительное» — «Черновик письма» — «Подходит» (вакансия переходит во вкладку «Подходят»)
7. **Отклик:** «Отклик в браузере» или `npm run hh-apply-chat -- --id=...`
8. Повторять шаги 3-7 по мере появления новых вакансий

### Docker-аналог рабочего цикла

Если используете Docker:
