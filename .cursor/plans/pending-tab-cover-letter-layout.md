# План: режим «На проверке» — кнопки сопроводительного и отклонения

## Текущее состояние (код)

### Разметка карточки (`dashboard/public/index.html`, `#card-tpl`)

1. **`.apply-chat-row`** — кнопка «Отклик в браузере» (`.btn-apply-chat`). В `app.js` она **включается**, если есть утверждённое письмо (`coverLetter.status === 'approved'` и непустой `approvedText`) — строки ~683–709.

2. **`.cover-draft-btn`** — «Черновик письма», показывается при `cl.status === 'pending'` и наличии вариантов (`variants.length`) — ~642–645.

3. **`.btn-regenerate-letter` / `.btn-view-approved`** — ветки `declined` / `approved` у письма.

4. **`.actions`** (для `item.status === 'pending'`, `actions.hidden = false` — ~714–715):
   - Сверху **`<textarea class="reason">`** — «причина…».
   - **`.actions-toolbar`**: слева иконка обновления + «Создать сопроводительное», справа «Подходит» и «Не подходит».

### Логика (`dashboard/public/app.js`, `renderCard`)

- Обработчики «Создать сопроводительное», refresh, `approve`/`reject` вешаются только при `item.status === 'pending'`.
- При refresh сейчас отключаются **и** `ok`, **и** `bad`, и поля cover (~752–756) — это **противоречит** требованию «причина + Не подходит всегда активны» — при реализации ослабить: например отключать только генерацию/«Подходит», но не `textarea` и `.bad`.

### Стили

- `.apply-chat-row`, `.cover-draft-btn` (отступ сверху у черновика), `.actions-toolbar` — `dashboard/public/style.css` (~630–693).

## Целевое поведение (только вкладка «На проверке», `status === 'pending'`)

| Элемент | Поведение |
|--------|-----------|
| **«Отклик в браузере»** | **Не показывать** на карточках со статусом вакансии `pending`, даже если письмо уже утверждено в данных (на «На проверке» отклик в браузере не нужен). |
| **«Создать сопроводительное»** | На месте текущей primary-кнопки в строке, где сейчас «Отклик в браузере» (объединить в одну горизонтальную зону с иконкой ↻ при необходимости). |
| **«Черновик письма»** | Справа от «Создать сопроводительное», **появляется** после генерации (как сейчас по условию `pending` + `variants.length`). |
| **«Подходит»** | Ещё правее, **появляется** вместе с черновиком (после появления вариантов письма), логика `send('approve')` без изменений по API. |
| **Причина отклонения** | В **самом низу** блока действий карточки. |
| **«Не подходит»** | Сразу **под** полем причины; **всегда видны**; по возможности **всегда активны** (не блокировать на время refresh/генерации, кроме краткого состояния во время самого `send`, если оставить как сейчас). |

### Вкладки «Подходят» / «Отклонённые»

- Карточки с `item.status !== 'pending'` **не трогать** по смыслу «Отклик в браузере»: для **утверждённых** записей по-прежнему показывать `.btn-apply-chat`, когда есть утверждённое письмо (текущая логика ~683–709), иначе пользователь потеряет отклик из дашборда.

## Изменения по файлам

### 1. `dashboard/public/index.html`

- Ввести явную обёртку для верхней строки режима проверки, например **`.pending-top-actions`** внутри или вместо текущей `.apply-chat-row` для единого flex-ряда: `[↻] [Создать сопроводительное] [Черновик письма] [Подходит]`.
- Перенести **из** `.actions-toolbar` в эту строку: иконку refresh, `.btn-cover` (одна кнопка «Создать…»), `.cover-draft-btn`, `.ok`.
- В **`.actions`** оставить только нижний блок: `textarea.reason` и кнопку `.bad` (порядок: сначала textarea, под ней «Не подходит»), либо вынести в `.pending-reject-row` для ясности в CSS.

Вариант без дублирования id: один шаблон, видимость секций правится из JS через классы/`hidden` по `item.status`.

### 2. `dashboard/public/app.js` — `renderCard`

- После клонирования шаблона для **`item.status === 'pending'`**:
  - Скрыть `.btn-apply-chat` (или не рендерить обработчик отклика для `pending`).
- Для **`item.status !== 'pending'`**:
  - Не показывать новую pending-only строку (если разделили в разметке), оставить текущее поведение отклика.
- Переназначить селекторы: `coverBtn`, `refreshBtn`, `ok` — на новые узлы в верхней строке; `ta`, `bad` — внизу.
- Условие показа **«Черновик письма»** и **«Подходит»**: как сейчас — при наличии черновика с вариантами (`cl?.status === 'pending' && (cl?.variants||[]).length`); при необходимости добавить показ «Подходит» синхронно с черновиком (уже следует из одного условия).
- Обновить **`refreshBtn` handler**: не ставить `bad.disabled = true` / не блокировать `textarea`, если по ТЗ отказ должен быть доступен всегда; при генерации письма — по желанию оставить блокировку только `ok`/cover на время запроса.

### 3. `dashboard/public/style.css`

- Flex для верхней строки: выравнивание по центру по вертикали, перенос на узких экранах (`flex-wrap`, `gap`).
- Убрать лишний верхний margin у `.cover-draft-btn`, если кнопка в одной строке с «Создать…».
- Стили для нижнего блока (отступ между верхним рядом и полем причины).

## Проверка (ручная)

1. Вкладка «На проверке»: карточка без письма — видны «Создать…», поле причины, «Не подходит»; **нет** «Отклик в браузере».
2. После генерации — в одной строке «Создать…», «Черновик письма», «Подходит»; снизу причина + «Не подходит».
3. Refresh текста вакансии — «Не подходит» и textarea остаются доступны (если так зафиксировано в коде).
4. Вкладка «Подходят»: при утверждённом письме — «Отклик в браузере» по-прежнему работает.
