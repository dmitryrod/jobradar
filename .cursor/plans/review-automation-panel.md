# План v2: автоматизация score и генерации писем для «Сбор вакансий» + «На проверке»

## Ревью кода

### Что в предыдущем плане было неудачно

1. **Одна панель только на «На проверке» не совпадает с реальным UX-сценарием.**
   Сейчас вкладка `collect` уже содержит рабочую форму запуска harvest (`dashboard/public/index.html`, `#harvest-panel`; `dashboard/public/app.js`, `fillHarvestFormFromApi()`, `btn-harvest-start`). Настройки, влияющие на **новые вакансии**, логичнее держать рядом со «Старт поиска».

2. **«Количество вариантов письма» зашито не только в бэкенде, но и во фронтенде.**
   В `lib/cover-letter-openrouter.mjs` промпт требует ровно 3 варианта, а в `dashboard/public/app.js` модалка `openDraftModal()` тоже жёстко делает `while (variants.length < 3)` и `splice(3)`. Значит, менять нужно **оба конца**, иначе UI сломает лишние варианты.

3. **Сохранять настройки пока некуда.**
   Есть только `GET /api/preferences` (`scripts/dashboard-server.mjs`), но нет `POST/PATCH`. Без этого панель будет декоративной.

4. **Батч по pending нельзя реализовывать клиентским циклом по карточкам.**
   `load()` рендерит только текущий список на вкладке, а генерация писем и смена статусов должны идти серверно, по данным из очереди. Иначе будут гонки, повторные рендеры и зависимость от открытого браузера.

5. **Опция `pending_only` как единый глобальный enum неудобна, если UI делится между двумя вкладками.**
   Для пользователя это не одна «магическая настройка», а два разных сценария:
   - поведение **для новых вакансий** во время/после сбора;
   - ручная обработка **уже накопившихся pending**.

---

## Зафиксированные решения

1. **Один порог:** `targetScore` используется и для auto-reject (`<`), и для auto-letter (`>=`).
2. **Нет валидного score:** автоматику не применять.
3. **Auto-reject:** `feedbackReason` оставлять пустым.
4. **Ручной батч по pending:** без подтверждения и без лимита записей за запуск.
5. **Очередь к LLM:** строго **по одной** вакансии за раз.
6. **Размещение UI:** часть настроек для новых вакансий в `Сбор вакансий`, сценарий для накопившихся pending и ручной запуск — в `На проверке`.

---

## Оптимизированная продуктовая модель

### Вместо одной перегруженной панели — две связанные поверхности

#### 1. Вкладка `Сбор вакансий`

Показывает **persistent-настройки**, которые влияют на новые карточки, попадающие из harvest:

- `Целевой Score`
- `Автоматическое отклонение`
- `Автоматическое сопроводительное письмо`
- `Условия генерации письма`:
  - `Для новых вакансий`
  - `Для новых вакансий и находящихся на проверке`
- `Количество вариантов сопроводительного письма` (только если включено авто-письмо)

Это место логично, потому что настройки влияют на то, что будет происходить **после «Старт поиска»**.

#### 2. Вкладка `На проверке`

Показывает **операционный блок** для накопившихся pending:

- компактное отображение текущих сохранённых настроек (`targetScore`, auto-reject, auto-letter, variants);
- опцию/метку, что ручной запуск работает по pending-карточкам без черновика;
- кнопку:
  - `Сгенерировать черновики для pending`

Кнопка запускает серверный проход по всем записям со статусом `pending`, у которых нет черновика письма, и обрабатывает их **по одной**.

### Почему так лучше

- не дублируем весь form-state в двух местах;
- всё, что относится к **новым** вакансиям, живёт рядом со сбором;
- всё, что относится к **накопившимся pending**, живёт рядом с вкладкой `На проверке`;
- при этом настройки общие и хранятся в одном месте.

## Контракт настроек

Расширить `config/preferences.json` секцией:

```json
"reviewAutomation": {
  "targetScore": 70,
  "autoRejectBelowTarget": false,
  "autoCoverLetterAtOrAboveTarget": false,
  "coverLetterScope": "new_only",
  "coverLetterVariantCount": 3
}
```

### Значения `coverLetterScope`

- `new_only` — авто-письмо только для новых вакансий из harvest;
- `new_and_pending` — авто-письмо для новых вакансий и разрешение на серверный проход по pending;
- `pending_only` — **не хранить как основной collect-mode**, а использовать как **внутренний режим ручного батча** на вкладке `На проверке`.

### Важная оптимизация

Во внешнем UI для вкладки `Сбор вакансий` пользователю не нужен отдельный выбор `pending_only`.
На практике это не collect-настройка, а **операция на backlog**. Поэтому:

- в `Сбор вакансий` сохраняем только `new_only` / `new_and_pending`;
- в `На проверке` кнопка запускает обработку pending как отдельный action, не ломая collect-настройку.

## Точное поведение

### A. Новые вакансии (после harvest)

После добавления новой записи в очередь:

1. Берём `scoreOverall`, если это валидное число.
2. Если score нет — ничего автоматически не делаем.
3. Если включено `autoRejectBelowTarget` и `score < targetScore`:
   - переводим запись в `rejected`;
   - `feedbackReason = ''`.
4. Иначе, если включено `autoCoverLetterAtOrAboveTarget` и `score >= targetScore`:
   - если `coverLetterScope === 'new_only'` или `new_and_pending`, генерируем черновик;
   - генерация идёт один раз, если черновика ещё нет.

### B. Накопившиеся `pending`

На вкладке `На проверке` кнопка запускает серверный проход:

1. Берём все записи `status === 'pending'`.
2. Пропускаем записи:
   - без валидного score;
   - со score `< targetScore`;
   - с уже существующим черновиком (`coverLetter.status === 'pending'` и есть `variants`);
   - с уже утверждённым письмом.
3. Для остальных по одной генерируем `Черновик письма`.
4. UI показывает результат батча: сколько обработано, сколько пропущено, сколько ошибок.

### C. Что делать с `new_and_pending`

`new_and_pending` трактуем так:

- **новые вакансии** обрабатываются автоматически после harvest;
- **накопившиеся pending** не стартуют «сами по таймеру», а доступны через кнопку ручного запуска на вкладке `На проверке`.

Это важная оптимизация: не делать скрытый фоновый LLM-проход без явного действия пользователя.

### 3. Бэкенд: интеграция с harvest

#### Файл

#### Что сделать

Важно: порядок именно такой, чтобы письмо не генерировалось для карточки, которая должна уйти в `Отклонённые`.

### 4. Бэкенд: ручной батч для pending

#### Файл

- `scripts/dashboard-server.mjs`

#### Новый endpoint

- `POST /api/review-automation/run-pending-cover-letters`

#### Поведение

- серверно читает очередь;
- обрабатывает записи строго **по одной**;
- не требует подтверждения и лимита;
- возвращает JSON с итогом:

```json
{
  "ok": true,
  "processed": 12,
  "generated": 7,
  "skipped": 5,
  "errors": []
}
```

## Декомпозиция работ

### Этап 1. Контракт и хранение

- defaults для `reviewAutomation`;
- read/write API;
- валидация;
- ручная проверка, что `preferences.json` не ломается.

### Этап 2. Серверная автоматика

- общий модуль `review-automation`;
- интеграция auto-reject и auto-cover-letter для новых записей;
- ручной pending-batch endpoint.

### Этап 3. Динамическое количество вариантов

- генератор LLM;
- нормализация массива;
- модалка черновика без жёсткой тройки.

### Этап 4. UI на `Сбор вакансий`

- форма настроек;
- загрузка/сохранение;
- условный показ полей.

### Этап 5. UI на `На проверке`

- summary-панель;
- кнопка запуска батча;
- тосты/индикаторы выполнения.

### Этап 6. Проверка сценариев

1. Новая вакансия со score ниже порога уходит в `Отклонённые`.
2. Новая вакансия со score выше порога получает черновик.
3. Вакансия без score не трогается.
4. Ручной батч на `На проверке` генерирует черновики только там, где их нет.
5. Вариантов письма приходит не 3, а столько, сколько задано в настройках.

## Риски

1. **Стоимость LLM** остаётся, но пользователь осознанно отказался от лимита.
2. **Гонки** между harvest и ручным батчем: перед генерацией всегда повторно проверять актуальное состояние записи.
3. **Дублирование UI** между `collect` и `pending` нужно минимизировать: одна каноническая модель настроек, а не две независимые формы.

---

## Итоговая рекомендация

Оптимальный вариант не «одна огромная панель в одном месте», а:

- **канонические настройки** для новых вакансий на вкладке `Сбор вакансий`;
- **операционный запуск для backlog pending** на вкладке `На проверке`;
- **вся автоматика серверная**, без клиентских циклов по карточкам;
- **динамическое число вариантов** как отдельная техническая подзадача, потому что сейчас hardcode сидит и в prompt, и в modal UI.
