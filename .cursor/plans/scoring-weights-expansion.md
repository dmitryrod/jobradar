# План: расширение скоринга вакансий (город, формат работы, доп. сигналы)

## Контекст (как сейчас)

- **LLM** (`lib/openrouter-score.mjs`) возвращает `scoreVacancy`, `scoreCvMatch`, опционально `scoreOverall`. Текст промпта задаёт смысл осей; обратная связь по отклонениям подмешивается в промпт.
- **Фолбэк итога**, если модель не дала валидный `scoreOverall`:  
  `scoreOverall = round(w_v * scoreVacancy + w_c * scoreCvMatch)`, веса из `preferences.json` → `llmScoreWeights` (`vacancy`, `cvMatch`), нормализация до суммы 1.
- **Жёсткие фильтры** до LLM (`lib/filters.mjs`): удалёнка/гибрид/офис по паттернам, зарплата. Они **не добавляют непрерывного града** в рейтинг — вакансия либо прошла, либо отсечена.

Итого: «рейтинг» в основном **одна шкала от модели** + редко **линейная смесь двух чисел**. Два коэффициента влияют только на фолбэк, а не на разброс верхней границы.

## Почему много вакансий на одном максимуме (например 88)

Вероятные причины (могут суммироваться):

1. **Калибровка модели** — склонность к «хорошим но не 100» оценкам (типичный потолок 85–92).
2. **Одна итоговая ось** — когда всё смешано в `scoreOverall`, модель сжимает различия.
3. **Округление** — `clampScore` даёт целые 0–100; мелкие отличия исчезают.
4. **Узкий постфильтр** — после фильтров остаётся однородный пул «уже неплохих» вакансий → LLM ставит близкие оценки.
5. **Два входа в фолбэк** — если `scoreVacancy` и `scoreCvMatch` часто близки, взвешенная сумма тоже мало дифференцирует.

Вывод: **увеличить число весов в `llmScoreWeights` без новых измеримых сигналов** частично поможет только если заставить модель стабильно возвращать **отдельные числовые оси**; иначе эффект будет слабым.

## Направления улучшения

### Вариант A — Расширить оси LLM + веса в `preferences.json` (рекомендуется как ядро)

Добавить в JSON-ответ модели поля, например:

- `scoreWorkFormat` (0–100) — насколько формат (удалёнка / гибрид / офис) совпадает с **явно заданными** предпочтениями кандидата в конфиге.
- `scoreLocation` (0–100) — город / релокация / «любая точка» vs твой **базовый город** или список допустимых локаций.
- Опционально: `scoreCompensation` (насколько вилка и бонусы выглядят привлекательно **относительно** порога и рынка — без дублирования жёсткого фильтра по `minMonthlyUsd`).

В промпт передавать структурированный блок из prefs, например:

```json
"candidatePreferences": {
  "baseCity": "Казань",
  "acceptableCities": ["Казань", "Удалённо"],
  "workFormatPreference": "remote_first",
  "hybridDaysMaxPerWeek": 2
}
```

**Формула итога (после парсинга):**

- Нормализованные веса `w_v, w_c, w_wf, w_loc, …` суммой 1.
- `scoreOverall = round( clamp( Σ_k w_k * score_k ) )` или оставить `scoreOverall` от модели только как один из компонентов / заменить на чистую агрегацию.

**Плюсы:** больше степеней свободы, проще объяснять в UI (подсказка по компонентам). **Минусы:** нужно обновить промпт, парсинг, дашборд, очередь; риск «галлюцинаций» по локации — лучше **подкреплять** детерминированным слоем (ниже).

### Вариант B — Детерминированный слой поверх LLM (сильно для разведения «одинаковых» 88)

После получения `scoreLlm` (или среднего по осям) вычислять **корректировки** из уже распарсенных полей вакансии (`parseVacancyPage` / адрес / условия):

| Сигнал | Пример правила |
|--------|----------------|
| Формат работы | Классификация: `remote` / `hybrid` / `office` / `unknown` по тем же/расширенным паттернам, что и фильтры. Бонусы/штрафы из prefs: `bonusRemote`, `penaltyOffice`, `bonusHybridIfAllowed`. |
| Город | Нормализация строки города из карточки; совпадение с `baseCity` или вхождение в `acceptableCities`; иначе штраф `penaltyOtherCity`. Для «релокация возможна» — отдельный маленький бонус или нейтрально, по твоей политике. |
| Зарплата | Уже есть оценка USD; мягкий бонус за запас над `minMonthlyUsd` (например логистическая кривая), не ломая фильтр. |

Итог:

`finalScore = clamp( scoreLlm + Σ adjustments )` или  
`finalScore = clamp( α * scoreLlm + (1-α) * scoreRuleBased )`.

**Плюсы:** предсказуемость, меньше кластеризации на одном числе от LLM. **Минусы:** нужны аккуратные пороги и тесты на краевые случаи (пустой город, смешанные формулировки).

### Вариант C — Разведение без смены «философии» скора

- Хранить **дробный** внутренний скор (например 0–1000 или одно знак после запятой) для сортировки; в UI показывать округлённый.
- Вторичный ключ сортировки: `maxUsd`, дата публикации, `vacancyId`.
- Лёгкая **калибровка**: линейное растяжение или перцентиль внутри батча (осторожно с интерпретацией).

## Расширение `preferences.json` (черновик схемы)

```json
{
  "llmScoreWeights": {
    "vacancy": 0.25,
    "cvMatch": 0.45,
    "workFormat": 0.15,
    "location": 0.15
  },
  "scoringGeo": {
    "baseCity": "",
    "acceptableCities": [],
    "remoteIsAcceptable": true,
    "penaltyUnknownLocation": -3
  },
  "scoringWorkFormat": {
    "prefer": "remote_first",
    "remoteBonus": 0,
    "hybridBonus": 0,
    "officePenalty": -5,
    "useRuleBasedAdjustment": true
  },
  "scoringSalarySoft": {
    "enabled": true,
    "maxBonusPoints": 8,
    "referenceUsd": 4000
  }
}

Числа бонусов/штрафов — в **пунктах** к итогу 0–100 или как доли отдельного слоя — зафиксировать в реализации одним способом.

## Порядок внедрения (предлагаемый)

1. **Инвентаризация полей** — что уже есть в записи очереди после парса (`address`, `workConditionsLines`, и т.д.); хватает ли для правил города/формата.
2. **Вариант B (минимальный)** — классификатор формата + город + мягкая зарплата; поля в prefs; функция `applyScoreAdjustments(parsed, prefs, llmScores)`; unit-тесты на строки.
3. **Вариант A** — расширить промпт и JSON; обновить `resolveThreeScores` → обобщённая агрегация N компонентов; дашборд: подсказка с разбивкой.
4. **Сортировка** — при равенстве `scoreOverall` использовать вторичные ключи + опционально «скрытый» дробный хвост.

## Риски

- Двойной учёт: то же самое и в фильтре, и в бонусе LLM, и в правилах — нужна явная иерархия («фильтр жёсткий, бонус только сверх порога»).
- Город на hh: опечатки и «МСК» vs «Москва» — нормализация или доверять только LLM для `scoreLocation` и правилам для очевидных случаев.
- Обратная совместимость очереди: старые записи без новых полей — дефолты.

## Дополнительные идеи

- **Теги от модели** (`tags[]`) уже есть — можно маппить теги на микро-бонусы (например «red_flag_contractor» → −10).
- **Персональные веса по роли**: два профиля весов в prefs и выбор по типу вакансии (эвристика по ключевым словам в title) — для продвинутых.
- **A/B**: логировать сырые компоненты в JSON очереди для анализа, почему много 88.

---

**Итог:** узкие `vacancy`/`cvMatch` сами по себе не объясняют «все на 88»; главный рычаг — **добавить измеримые оси** (LLM +/или правила) и **не полагаться на один округлённый `scoreOverall` без вторичной сортировки**. Оптимальная связка для твоего кейса: **B + частично A** (правила для города/формата, LLM для `scoreLocation` только если парсер не уверен).
