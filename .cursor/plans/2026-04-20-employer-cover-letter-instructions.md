# План: учёт инструкций работодателя в сопроводительном письме

## Summary
Добавляем в пайплайн вакансии отдельное структурированное поле с инструкциями работодателя к отклику, чтобы скрипт не терял важные требования из текста вакансии, подстраивал формат письма под эти требования и при необходимости легально выходил за обычный лимит длины.

## Situation Snapshot
Что уже есть:
- Текст вакансии парсится в `lib/vacancy-parse.mjs`, а в очередь сохраняются `descriptionForLlm`, `vacancyDescriptionFull` и `hhWorkConditions` через `scripts/harvest.mjs`.
- Скоринг вакансии делается в `lib/openrouter-score.mjs`.
- Генерация сопроводительного письма делается в `lib/cover-letter-openrouter.mjs`.
- Ручная и автоматическая генерация писем идёт через `scripts/dashboard-server.mjs` и `lib/review-automation.mjs`.
- Во фронтенде уже есть место для отображения рисков, тегов и деталей карточки: `dashboard/public/index.html`, `dashboard/public/app.js`.

Что болит:
- Сейчас нет отдельного поля под инструкции работодателя, поэтому пожелания вида "начните письмо со слов ...", "приложите GitHub", "ответьте на вопрос", "распишите тестовое" не выделяются и не получают приоритет.
- Генерация письма опирается на `descriptionForLlm`, а не на гарантированно полный и структурированный набор требований.
- Правило "короткое письмо" сейчас жёстко прошито в промпт и конфликтует с вакансиями, где работодатель явно просит длинный или структурированный ответ.

Что нельзя трогать:
- Текущий flow `harvest -> queue -> dashboard -> cover-letter` должен остаться совместимым.
- Нельзя ухудшить текущую генерацию для обычных вакансий без специальных инструкций.
- Нельзя ломать автоматизацию review/pending и существующие поля очереди.

Constraints:
- Источник требований — именно свободный текст описания вакансии на hh.ru; отдельных специальных полей "как откликаться" нет и на них нельзя рассчитывать.
- Формулировки очень разные: от одного кодового слова до набора обязательных ответов, ссылок и мини-скрининга.
- Нужна устойчивость к обрезке текста и ложным срабатываниям.

## Примеры паттернов, которые надо поддержать
1. Обязательная фраза в начале письма.
2. Обязательные ссылки: GitHub, портфолио, Telegram, проекты.
3. Обязательные вопросы работодателя с короткими ответами.
4. Обязательный перечень кейсов/результатов/ROI.
5. Обязательный мини-скрининг по пунктам требований (`+` / `-`).
6. Сигнал "отправьте только резюме" или "короткое письмо".
7. Сигнал "можно/нужно письмо длиннее обычного".
8. Инструкция, спрятанная прямо внутри общего описания вакансии обычным абзацем, без отдельного визуального блока hh.ru.

## Варианты реализации

### Вариант A — только эвристики в генераторе письма
Идея:
- В `lib/cover-letter-openrouter.mjs` перед генерацией пробегаться по `vacancyDescriptionFull` regex-эвристиками.
- Если нашли триггеры вроде `укажите`, `приложите`, `в сопроводительном письме`, `начните со слов`, `ответьте на вопрос`, перестраивать prompt на лету.

Плюсы:
- Быстро внедряется.
- Не требует менять схему queue и harvest.

Минусы:
- Логика размазывается по генератору письма.
- Нет отдельного поля в карточке и в queue.
- Один и тот же expensive parsing придётся делать заново при каждой генерации.
- Плохо масштабируется на сложные кейсы вроде screen-question/checklist.

### Вариант B — отдельный LLM-extractor только в момент генерации письма
Идея:
- Перед `generateCoverLetterVariants()` делать короткий extraction-pass: "вытащи инструкции работодателя в JSON".
- Затем этот JSON подавать в основной prompt генерации.

Плюсы:
- Лучше качество, чем чистые regex.
- Не надо трогать harvest/scoring.

Минусы:
- Два LLM-вызова на каждую генерацию письма.
- Нет предрасчёта в queue и нет UI-прозрачности до генерации.
- Автоматизация pending/new будет дороже и медленнее.

### Вариант C — рекомендованный: structured extraction на этапе harvest/refresh + использование в генерации
Идея:
- На этапе `scoreVacancyWithOpenRouter()` или рядом с ним извлекать структурированные `employerInstructions`.
- Сохранять их в queue.
- Показывать в карточке и использовать как обязательный блок при генерации письма.

Плюсы:
- Один раз извлекли — потом переиспользуем в scoring, UI и letter generation.
- Пользователь видит инструкции заранее и может быстро проверить корректность.
- Подходит и для ручной генерации, и для review-automation.
- Проще дальше добавить отдельный сигнал в сортировку/приоритет.

Минусы:
- Требует смены контракта записи в queue и refresh-flow.
- Нужна аккуратная миграция и fallback на старые записи.

## Recommendation
Рекомендую **Вариант C** с лёгким hybrid-слоем:
- сначала дешёвые эвристики для детекта "похоже, тут есть instructions";
- затем один LLM extraction-pass в рамках существующего scoring-вызова;
- результат сохранять в queue как отдельный JSON-блок;
- генератор письма использовать этот JSON как источник истины;
- если structured block отсутствует (старые записи), делать fallback на эвристики + full description.

Это даёт лучший баланс по качеству, стоимости и прозрачности.

## Предлагаемый контракт данных

```json
{
  "employerInstructions": {
    "detected": true,
    "confidence": 0.92,
    "strictness": "mandatory",
    "responseFormat": "question_answer",
    "lengthPolicy": "extended_if_needed",
    "mustStartWith": "Я тот специалист, который вам нужен",
    "mustMention": [
      "опыт с n8n",
      "примеры API-интеграций",
      "почему интересна роль"
    ],
    "mustAnswerQuestions": [
      "Какой самый быстрый MVP вы собирали и за какой срок?",
      "Какие AI-инструменты использовали в работе?"
    ],
    "requiredArtifacts": [
      { "type": "github", "required": true },
      { "type": "portfolio", "required": false },
      { "type": "telegram", "required": false }
    ],
    "screeningChecklist": [],
    "rawFragments": [
      "В сопроводительном письме, пожалуйста, начните со слов ..."
    ],
    "notesForGenerator": "Если работодатель просит ответы на вопросы, письмо можно сделать длиннее обычного."
  }
}
```

### Поля, которые реально нужны v1
- `detected`
- `confidence`
- `strictness`: `none | desired | mandatory`
- `responseFormat`: `plain_short | plain_extended | question_answer | checklist | resume_only | mandatory_phrase_first`
- `lengthPolicy`: `short | normal | extended_if_needed`
- `mustStartWith`
- `mustMention[]`
- `mustAnswerQuestions[]`
- `requiredArtifacts[]`
- `screeningChecklist[]`
- `rawFragments[]`
- `notesForGenerator`

## Где менять код

### 1. Извлечение и сохранение
- `lib/openrouter-score.mjs`
  - расширить JSON-ответ модели новым блоком `employerInstructions`;
  - добавить в prompt явную задачу: вычитывать из свободного текста описания вакансии инструкции работодателя к отклику и отделять их от обычного описания роли/стека/условий;
  - модель должна считать такие фрагменты высокоприоритетными, даже если они находятся в хвосте длинного текста.
- `scripts/harvest.mjs`
  - сохранять `employerInstructions` в `record`;
  - сохранить также флаг/short badge для UI, например `hasEmployerInstructions`.
- `scripts/dashboard-server.mjs`
  - при `refresh-body` пересобирать `employerInstructions`, как и score.

### 2. Генерация письма
- `lib/cover-letter-openrouter.mjs`
  - использовать `record.employerInstructions` как обязательный блок prompt;
  - заменить жёсткое правило `4–8 предложений` на динамическую политику длины;
  - если `responseFormat = question_answer` или `checklist`, разрешать более длинное письмо и более структурированный ответ;
  - если есть `mustStartWith`, enforce first sentence exactly.

### 3. UI
- `dashboard/public/index.html`
  - добавить компактный блок "Инструкции работодателя" в карточке.
- `dashboard/public/app.js`
  - рендерить badge/summary:
    - "Есть обязательные инструкции"
    - "Нужен GitHub"
    - "Нужно ответить на 2 вопроса"
    - "Письмо можно длиннее"

### 4. Автоматизация
- `lib/review-automation.mjs`
  - при автогенерации использовать тот же structured block без дополнительных special-case веток;
  - опционально в будущем добавить настройку "автогенерировать только если employerInstructions.confidence >= X".

## Как менять prompt генерации

### Сейчас
Сейчас prompt в `lib/cover-letter-openrouter.mjs` просит "короткое письмо" и держит формат слишком жёстко.

### Что нужно вместо этого
1. Базовый режим по-прежнему короткий.
2. Если работодатель не дал специальных инструкций, ничего не меняем.
3. Если `strictness = mandatory`, инструкции идут выше anti-AI style rules по приоритету.
4. Если формат ответа special (`question_answer`, `checklist`), генератор строит письмо под этот формат.
5. Если `lengthPolicy = extended_if_needed`, разрешить длину выше обычной, но только ради выполнения требований работодателя, а не ради "воды".

### Нормальная логика длины
- `plain_short` -> 4–8 предложений
- `plain_extended` -> 8–14 предложений
- `question_answer` -> короткое intro + ответы по пунктам
- `checklist` -> intro + список `+ / -` или компактные строки по требованиям
- `resume_only` -> очень короткий ответ, без лишней мотивации

## Как лучше извлекать инструкции работодателя

### Этап 1. Эвристический pre-detect
Поймать вакансии, где почти наверняка есть instructions прямо внутри текста описания:
- `в сопроводительном письме`
- `как откликнуться`
- `обязательно укажите`
- `приложите`
- `напишите`
- `ответьте`
- `начните со слов`
- `отклики без`
- `при отклике`
- `ключевой вопрос`

Если ничего не найдено, extraction можно всё равно делать, но моделью с дешёвым компактным JSON и низким лимитом.

### Этап 2. LLM structuring
Модель должна отвечать не общим summary, а именно вычленять:
- что обязательно;
- что желательно;
- какой нужен формат ответа;
- какие вопросы работодатель хочет увидеть;
- какие ссылки/артефакты нужно приложить;
- можно ли выйти за длину стандартного письма;
- где именно в исходном описании найден этот фрагмент, чтобы его можно было показать в UI и проверить глазами.

### Этап 3. Fallback
Если `employerInstructions.detected = false`, генератор работает как сейчас.

## Нужно ли менять scoring
Для v1 я бы **не смешивал** это с `scoreOverall` напрямую.

Почему:
- инструкции работодателя влияют в первую очередь на **качество отклика**, а не на **релевантность вакансии**;
- если смешать это сразу в общий score, можно случайно занизить хорошие вакансии только потому, что там сложный screening.

Но стоит добавить:
- `instructionComplexity`: `none | low | medium | high`
- `instructionFitRisk`: текстовый сигнал в `risks`

А уже v2 можно добавить:
- отдельный secondary sort boost для вакансий, где требования к письму можно качественно закрыть;
- или penalty, если нужна информация, которой у кандидата точно нет.

## Top Risks
1. **Ложные срабатывания extraction** — модель может принять обычные требования вакансии за инструкции к отклику.
2. **Слишком агрессивное удлинение письма** — письма начнут раздуваться без реальной необходимости.
3. **Поломка совместимости старых записей** — в queue будут записи без нового поля.

## Mitigations
- Показывать `rawFragments` в UI, чтобы можно было глазами проверить источник.
- Делать удлинение письма только если `responseFormat != plain_short` или `lengthPolicy = extended_if_needed`.
- Везде держать fallback на старый режим генерации.
- В генерации использовать не только `descriptionForLlm`, а при наличии инструкций обязательно подтягивать релевантный фрагмент из `vacancyDescriptionFull`, чтобы хвост описания не терялся из-за обрезки.
